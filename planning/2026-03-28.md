# 2026-03-28 修改紀錄

## Scheduler 面板 — Timeframe 切換 UI

### 背景

5m 模型部署後，若 scheduler 仍設定為 `1m`，它會載入 `ppo_BTCUSDT_1m.zip` 並抓 1m K 棒，與 5m 模型不符。
Timeframe 設定存於 `runtime/scheduler.timeframes`（預設 `1m`），API 早已實作，但 Admin UI 沒有對應控制項。

### 修改內容

#### `app/admin/page.py` — Scheduler 面板加入 Active Timeframes

**HTML（scheduler 面板）**
- 在 Active Symbols 下方新增 `Active Timeframes` 區塊
- `<div id="scheduler-timeframe-pills" class="toggle-pill-group">` 作為 pill 容器

**JS — 初始化（refreshAll 內）**
- 呼叫 `GET /scheduler/timeframes` 取得 `supported_timeframes` 與 `timeframe_names`
- 按 tfOrder（1m/3m/5m/… 時間順序）排序後渲染 pill
- 目前 active timeframes 預設 selected；點選 pill 時標記 `_schedulerPillsDirty = true`

**JS — 儲存（`scheduler-strategy-save` action）**
- Apply Strategy State 按鈕觸發時，額外收集 `scheduler-timeframe-pills` 中被選取的 pill
- 若有選取，呼叫 `POST /scheduler/timeframes` with `{ timeframe_names: [...] }`
- 與 symbols / strategy 同步儲存

### 使用方式

部署 5m 模型時的完整步驟：
1. Training 頁面 → 點「Deploy」→ 複製模型成 `ppo_BTCUSDT_5m.zip`
2. Admin → Scheduler 面板 → 取消選取 `1m`，點選 `5m`
3. 按 **Apply Strategy State** 儲存
4. 重啟 scheduler（Start 按鈕）

## 5m Episode Length 修正

### 問題

原本 trainer 將以下 bar 數寫死：

- `TRAIN_EP_LEN = 1440`
- `EVAL_EP_LEN = 2880`

這對 `1m` 等於 `1d / 2d`，但對 `5m` 會變成 `5d / 10d`，導致 `1m` 與 `5m` 評估不公平。

### 修正

- `app/training/ppo_trainer.py`
  - 新增 `resolve_episode_lengths(timeframe)`
  - 依 timeframe 回傳較公平的 episode 長度

目前規則：

- `1m` → `train_ep_len=1440`, `eval_ep_len=2880`
- `5m` → `train_ep_len=576`, `eval_ep_len=1152`

### 額外調整

- `app/api/main.py`
- `app/admin/page.py`

training job details 會同步記錄：

- `train_ep_len`
- `eval_ep_len`
