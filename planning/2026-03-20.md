## 二、Broker Adapter Groundwork

### 背景

`SimulatedLiveExecutionAdapter` 之前繼承 `PaperExecutionAdapter`，實質上是空殼；缺乏可插拔的 broker 抽象層。

### 新增 `app/execution/live_broker.py`

定義 `BrokerClient` Protocol 與 `SimulatedBrokerClient`：

```python
class BrokerClient(Protocol):
    broker_name: str
    def place_order(self, symbol, side, qty, ref_price) -> Dict[str, Union[str, float]]: ...

class SimulatedBrokerClient:
    broker_name = "simulated"
    def place_order(self, symbol, side, qty, ref_price):
        return {"status": "FILLED", "fill_price": ref_price, "fill_qty": qty}
```

提供完整 execution 函式（皆接受 `broker_client: BrokerClient`）：
- `execute_risk_event_id`
- `execute_latest_risk`
- `execute_pending_approved_risks`
- `execute_risk_event_ids`

### 修改 `app/execution/adapter.py`

- 所有 adapter 加入 `is_live: bool` 與 `placeholder: bool` class attributes
- `SimulatedLiveExecutionAdapter` 改為獨立 class，路由至 `live_broker` 模組
- `get_execution_backend_status()` 改用 `adapter.placeholder` 取代 `adapter.name == "simulated_live"` hardcode 判斷
- `_execution_backend_check()` 在 API 回傳中加入 `"is_live"` 欄位

### 新增測試（7 個）

- Simulated broker `place_order` 語意驗證
- `live_broker` direct call（`execute_risk_event_id`、skip duplicate、`execute_risk_event_ids`）
- `SimulatedLiveExecutionAdapter` end-to-end（`execute_pending_approved_risks`）
- `SimulatedLiveExecutionAdapter` class attributes 驗證

## 三、Broker / Order-Level Alerting

### 背景

`_pipeline_check` 缺少 `symbol`/`qty`/`latest_fill`；`_broker_protection_check` 沒有 unfilled order 偵測。

### 修改 `app/api/main.py`

**`_pipeline_check()`**

- `latest_order` SQL 擴充為 `side, symbol, qty, status, created_at`
- 新增 `latest_fill` 查詢（`fills` 最新一筆，含 `symbol/side/qty/price/created_at`）
- 新增 `unfilled_order_count` 查詢：

```sql
SELECT COUNT(*) FROM orders o
LEFT JOIN fills f ON f.order_id = o.id
WHERE f.id IS NULL AND o.status NOT IN ('CANCELLED', 'REJECTED', 'EXPIRED');
```

**`_broker_protection_check()`**

stale order 檢查後新增 unfilled order 條件：

```python
if unfilled_order_count > 0:
    result["status"] = "degraded"
    result["severity"] = "high"
    result["reason_code"] = "unfilled_orders_detected"
    result["reason"] = f"{unfilled_order_count} order(s) have no corresponding fill."
    result["recommended_action"] = "inspect_and_reconcile_orders"
    result["unfilled_order_count"] = unfilled_order_count
    return result
```

### 修改 `app/alerting/broker.py`

- `_build_fingerprint()` 加入 `"unfilled_order_count"`
- `maybe_send_broker_alert()` 訊息加入 `unfilled_orders=N` 與 `latest_fill_price=X`

### 新增測試（5 個）

- `_pipeline_check` 含 `latest_fill` 與 `unfilled_order_count`
- `_broker_protection_check` 在有 unfilled order 時回傳 `degraded`
- unfilled order 修復後 protection 回到 `ok`
- alert message 含 `unfilled_orders=N`

## 四、Worker Separation Convergence

### 背景 / 問題

1. **Double execution bug**：split-workers 模式下，scheduler 執行完整 pipeline，data/strategy/execution worker 也各自直接執行對應邏輯 → 工作重複做兩次。
2. **Worker crash 無回收**：job 卡在 `leased` 狀態後沒有自動恢復機制。
3. **False coupling**：`data-worker`、`strategy-worker`、`execution-worker` 有 `depends_on: api`，但實際只需要 DB。

### 修改 `app/core/job_queue.py`

新增 `reclaim_stale_leased_jobs(connection, lease_timeout_seconds)`：
- 查詢 `started_at` 超過 timeout 的 `leased` jobs
- 重設為 `queued`、清空 `started_at`

### 修改 `app/scheduler/runner.py`

- 新增 `_reclaim_stale_leases(pipeline_orchestration, queue_drain)` helper
- 每次 loop 開始時呼叫，僅在 queue-consuming 模式下有效

### 修改 `docker-compose.yml`

- `scheduler`：加入 `CRYPTO_PIPELINE_ORCHESTRATION` 與 `CRYPTO_JOB_LEASE_TIMEOUT_SECONDS` 環境變數，comment 說明 split-workers 應設 `queue_dispatch`
- `data-worker`、`strategy-worker`、`execution-worker`：command 改為 `--queue-drain`，移除 `depends_on: api`

### 修改 `app/core/settings.py`

加入 `JOB_LEASE_TIMEOUT_SECONDS = _get_int("CRYPTO_JOB_LEASE_TIMEOUT_SECONDS", 300)`

### 新增測試（3 個）

- stale lease 超時後被重設為 `queued`
- 未超時的 lease 不受影響
- `completed` / `failed` job 不受影響
