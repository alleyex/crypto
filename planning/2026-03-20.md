```python
class BrokerClient(Protocol):
    broker_name: str
    def place_order(self, symbol, side, qty, ref_price) -> Dict[str, Union[str, float]]: ...

    broker_name = "simulated"
```

## 四、Worker Separation Convergence

### 背景 / 問題

1. **Double execution bug**：split-workers 模式下，scheduler 執行完整 pipeline，data/strategy/execution worker 也各自直接執行對應邏輯 → 工作重複做兩次。
2. **Worker crash 無回收**：job 卡在 `leased` 狀態後沒有自動恢復機制。
3. **False coupling**：`data-worker`、`strategy-worker`、`execution-worker` 有 `depends_on: api`，但實際只需要 DB。

後續補註（2026-03-21）：
- split-workers 已再往前推進為 `data-worker`、`strategy-worker`、`risk-worker`、`execution-worker`
- pipeline 主路徑已正式固定為 `market_data -> strategy -> risk -> execution`

### 修改 `app/core/job_queue.py`

新增 `reclaim_stale_leased_jobs(connection, lease_timeout_seconds)`：
- 查詢 `started_at` 超過 timeout 的 `leased` jobs
- 重設為 `queued`、清空 `started_at`

### 修改 `app/scheduler/runner.py`

- 新增 `_reclaim_stale_leases(pipeline_orchestration, queue_drain)` helper
- 每次 loop 開始時呼叫，僅在 queue-consuming 模式下有效

### 修改 `docker-compose.yml`

- `scheduler`：加入 `CRYPTO_PIPELINE_ORCHESTRATION` 與 `CRYPTO_JOB_LEASE_TIMEOUT_SECONDS` 環境變數
- `data-worker`、`strategy-worker`、`execution-worker`：command 改為 `--queue-drain`，移除 `depends_on: api`

### 修改 `app/core/settings.py`

加入 `JOB_LEASE_TIMEOUT_SECONDS = _get_int("CRYPTO_JOB_LEASE_TIMEOUT_SECONDS", 300)`

### 新增測試（3 個）

- stale lease 超時後被重設為 `queued`
- 未超時的 lease 不受影響
- `completed` / `failed` job 不受影響

## 七、Execution Job 孤兒訂單偵測

### 修改 `app/pipeline/execution_job.py`

新增 `scan_orphan_orders(connection)`：
```sql
SELECT o.id, o.symbol, o.side, o.qty, o.status, o.created_at
FROM orders o
LEFT JOIN fills f ON f.order_id = o.id
WHERE f.id IS NULL
  AND o.status NOT IN ('CANCELLED', 'REJECTED', 'EXPIRED')
```

## 八、Multi-Strategy Double-Execution 修復

### 背景 / 問題

同一 pipeline cycle 中，兩個策略都對 BTCUSDT 發出 BUY 訊號。風控讀 `positions` 表，但 positions 在 execution 之前不更新。兩個策略都看到 `current_qty=0`，兩筆都被 APPROVED，兩筆都執行 → 超過 `max_position_qty`。

### 修改 `app/risk/risk_service.py`

新增 `SELECT_PENDING_APPROVED_BUY_COUNT_SQL`：
```sql
SELECT COUNT(*)
FROM risk_events re
LEFT JOIN orders o ON o.risk_event_id = re.id
WHERE re.symbol = ?
  AND re.signal_type = 'BUY'
  AND re.decision = 'APPROVED'
  AND o.id IS NULL;
```

新增 `_get_pending_approved_buy_qty(connection, symbol, order_qty)`：
- `orders` 表不存在時回傳 `0.0`（pre-migration 安全）
- 回傳 `count * order_qty`

在 `_evaluate_signal_row()` 中：
```python
pending_qty = _get_pending_approved_buy_qty(connection, symbol, order_qty)
effective_buy_qty = current_qty + pending_qty
```

BUY 的 position 檢查改用 `effective_buy_qty`，reason string 含 `pending_qty` 值供觀測。

### 新增測試（2 個）

- 同一 cycle 第二個策略 BUY → REJECTED，reason 含 `pending_qty`
- 第一個策略 order 落地後，第二個策略被 actual position 擋住（非 pending）

## 十五、Soak Validation 摘要改善

### 背景

`continuous_span_hours`（wall-clock span）在系統重啟或 degraded 期間仍持續累積，無法反映實際健康運行時數。

### 修改 `app/validation/soak_history.py`

新增常數與欄位：

| 新增項目 | 說明 |
|---|---|
| `SOAK_SNAPSHOT_INTERVAL_SECONDS = 60` | snapshot 間隔，與 scheduler `--interval` 對齊 |
| `SOAK_ACCUMULATED_TARGET_HOURS = 72` | 累積健康時數目標（低於 168h 以容忍開發機維護重啟） |
| `accumulated_ok_hours` | `ok_count × 60s / 3600` |
| `longest_ok_streak_hours` | 最長連續 ok 區間時數 |
| `ok_rate` | `ok_count / total_count`（四捨五入至小數第 4 位） |
| `meets_accumulated_target` | `accumulated_ok_hours >= 72` |
| `remaining_accumulated_hours` | `max(0, 72 - accumulated_ok_hours)` |
| `remaining_span_hours` | 原 `remaining_hours_to_target` 改名 |

現況（2026-03-20）：wall-clock 51.95h、accumulated ok 17.75h、ok rate 85.2%、最長 streak 4.28h；距 72h 目標還需約 54h。
